**Step 2: Create `docker/` directory and `docker/Dockerfile`.**

The Dockerfile implements a 3-stage multi-stage build per D-05:

```dockerfile
# в”Ђв”Ђ Stage 1: deps в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ
FROM node:22-alpine AS deps
WORKDIR /app

# corepack is included with Node.js 22; enables pnpm without network call
RUN corepack enable pnpm

# Copy lockfile and manifests first for maximum layer cache
# pnpm-workspace.yaml is required (blockExoticSubdeps: true)
# .npmrc is required (engine-strict=true)
COPY package.json pnpm-lock.yaml pnpm-workspace.yaml .npmrc ./

# Install ALL deps (devDeps needed for vite build)
RUN pnpm install --frozen-lockfile

# в”Ђв”Ђ Stage 2: builder в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ
FROM node:22-alpine AS builder
WORKDIR /app

RUN corepack enable pnpm

# Copy node_modules from deps stage
COPY --from=deps /app/node_modules ./node_modules

# Copy all source files (filtered by .dockerignore)
COPY . .

# Build the SvelteKit app вЂ” adapter-node outputs to build/
RUN pnpm run build

# в”Ђв”Ђ Stage 3: runtime в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ
FROM node:22-alpine AS runtime
WORKDIR /app

RUN corepack enable pnpm

# Install curl for HEALTHCHECK (not present in node:22-alpine by default)
RUN apk add --no-cache curl

# Copy package manifests for pnpm prune
COPY --from=builder /app/package.json ./
COPY --from=builder /app/pnpm-lock.yaml ./
COPY --from=builder /app/pnpm-workspace.yaml ./
COPY --from=builder /app/.npmrc ./

# Copy full node_modules, then prune to production-only
COPY --from=deps /app/node_modules ./node_modules
RUN pnpm prune --prod

# Copy built app (adapter-node output)
COPY --from=builder /app/build ./build

# Set ownership to non-root node user (uid 1000, already exists in node:22-alpine)
RUN chown -R node:node /app

# Run as non-root user per D-07
USER node

# Set production environment per D-11
ENV NODE_ENV=production
# Default port per D-09 (adapter-node default)
ENV PORT=3000

EXPOSE 3000

# Health check per D-08 вЂ” probes the existing /api/health endpoint
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:3000/api/health || exit 1

# adapter-node entry point вЂ” build/index.js imports instrumentation.server.js first
CMD ["node", "build/index.js"]
```

Key details:

- `node:22-alpine` per D-06 (matches project Node.js 22 LTS requirement)
- `pnpm install --frozen-lockfile` for reproducible installs
- `pnpm prune --prod` in runtime stage removes devDependencies
- `USER node` per D-07 (node user already exists in node:22-alpine at uid 1000)
- `HEALTHCHECK` per D-08 with curl
- `CMD ["node", "build/index.js"]` вЂ” instrumentation.server.js is auto-imported by the build output
- `.npmrc` and `pnpm-workspace.yaml` are copied alongside `package.json` and `pnpm-lock.yaml` to avoid `engine-strict` and `blockExoticSubdeps` failures

**Step 1: Create `docker/docker-compose.prod.yml`.**

Per D-02: includes app + postgres, NOT Aspire Dashboard. Per D-03: uses `env_file: .env` for all secrets.

```yaml
# Production compose вЂ” run with: docker compose -f docker/docker-compose.prod.yml up
#
# Requires a .env file at the project root with all variables from .env.example.
# IMPORTANT: DATABASE_URL must use the service name "postgres" (not "localhost")
# because containers communicate over the Docker network.
# Example: DATABASE_URL=postgres://postgres:postgres@postgres:5432/app

services:
  app:
    build:
      context: ..
      dockerfile: docker/Dockerfile
    ports:
      - '${PORT:-3000}:${PORT:-3000}'
    env_file: ../.env
    environment:
      - NODE_ENV=production
    depends_on:
      postgres:
        condition: service_healthy
    restart: unless-stopped

  postgres:
    image: postgres:17-alpine
    env_file: ../.env
    ports:
      - '5432:5432'
    volumes:
      - postgres_prod_data:/var/lib/postgresql/data
    healthcheck:
      test: ['CMD-SHELL', 'pg_isready -U $$POSTGRES_USER -d $$POSTGRES_DB']
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 10s
    restart: unless-stopped

volumes:
  postgres_prod_data:
```

Key details:

- `context: ..` because the Dockerfile is at `docker/Dockerfile` but the build context must be the project root (so COPY . . gets src/, package.json, etc.)
- `env_file: ../.env` вЂ” relative to the compose file location in docker/
- `depends_on` with `condition: service_healthy` ensures postgres is ready before app starts
- Volume name `postgres_prod_data` (distinct from dev `postgres_data`) to avoid data conflicts
- No hardcoded passwords anywhere вЂ” all via env_file per D-03 and D-04
- `restart: unless-stopped` for production resilience

<threat_model>

## Trust Boundaries

| Boundary                      | Description                                                      |
| ----------------------------- | ---------------------------------------------------------------- |
| Build context -> Image layers | Source files and potentially secrets cross into the image        |
| Host env -> Container runtime | Environment variables passed via env_file at runtime             |
| Container -> Container        | App container connects to postgres container over Docker network |

## STRIDE Threat Register

| Threat ID | Category               | Component                      | Disposition | Mitigation Plan                                                                                            |
| --------- | ---------------------- | ------------------------------ | ----------- | ---------------------------------------------------------------------------------------------------------- |
| T-08-01   | Information Disclosure | docker/Dockerfile              | mitigate    | .dockerignore excludes .env, .env.\*; secrets never COPYed into image; only passed via env_file at runtime |
| T-08-02   | Elevation of Privilege | docker/Dockerfile              | mitigate    | USER node directive in runtime stage; verified by `docker run --rm <image> whoami` outputting `node`       |
| T-08-03   | Information Disclosure | docker/docker-compose.prod.yml | mitigate    | No hardcoded passwords; all secrets via env_file: .env which is gitignored                                 |
| T-08-04   | Denial of Service      | docker/Dockerfile              | accept      | No resource limits (mem/cpu) in compose вЂ” out of scope for starter template; documented as next step       |
| T-08-05   | Tampering              | docker/Dockerfile              | mitigate    | --frozen-lockfile ensures reproducible dependency installation; no arbitrary package resolution            |

</threat_model>
