Create the directory `src/routes/api/health/` and the file `src/routes/api/health/+server.ts`.

IMPORTANT: Run `pnpm run check` first (via `svelte-kit sync`) to generate the `$types.ts` file for this new route. The `RequestHandler` type import in the file depends on this auto-generated file. If check fails because the route file doesn't exist yet, create the file first and then run check.

```typescript
// src/routes/api/health/+server.ts
import { json } from '@sveltejs/kit';
import type { RequestHandler } from './$types';
import { client } from '$lib/server/db';
import { logger } from '$lib/server/logger';

/**
 * GET /api/health
 *
 * Liveness probe for Docker health checks and load balancers.
 * Checks database connectivity using a raw SELECT 1 (no ORM overhead).
 *
 * Returns:
 *   200 { status: 'ok', db: 'ok', timestamp: ISO8601 }      — healthy
 *   503 { status: 'error', db: 'error', timestamp: ISO8601 } — DB unreachable
 */
export const GET: RequestHandler = async () => {
  const timestamp = new Date().toISOString();

  try {
    // Raw postgres.js tagged template — no Drizzle ORM (per D-18)
    await client`SELECT 1`;

    return json({ status: 'ok', db: 'ok', timestamp }, { status: 200 });
  } catch (err: unknown) {
    logger.error({ err }, 'Health check DB query failed');
    return json({ status: 'error', db: 'error', timestamp }, { status: 503 });
  }
};
```

After creating the file:

1. Run `pnpm run check` — TypeScript must compile cleanly. If `$types` is not yet generated, running check triggers `svelte-kit sync` which creates it.
2. Run `pnpm run lint` — ESLint must pass on the new file.
3. Run `pnpm run knip` — confirm no new unused export warnings. The health route is an entry point (SvelteKit file-based routing) so knip should not flag it.

If knip flags `src/routes/api/health/+server.ts` as unused (unlikely with SvelteKit plugin), add it to the `ignore` array in knip.config.ts. Do not add it to `ignoreDependencies`.
</action>

<threat_model>

## Trust Boundaries

| Boundary                      | Description                                           |
| ----------------------------- | ----------------------------------------------------- |
| Public internet → /api/health | Unauthenticated endpoint; no rate limiting in Phase 5 |
| Health check → PostgreSQL     | Raw SQL query crosses app-to-DB trust boundary        |
| Error response → Client       | DB error details must NOT leak in response body       |

## STRIDE Threat Register

| Threat ID | Category                                    | Component                        | Disposition | Mitigation Plan                                                                                                                                                 |
| --------- | ------------------------------------------- | -------------------------------- | ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| T-05-06   | Information Disclosure (Error leakage)      | src/routes/api/health/+server.ts | mitigate    | catch block logs err server-side via logger.error; response body only contains generic { status: 'error', db: 'error' } — no DB error message exposed to client |
| T-05-07   | Denial of Service (Health endpoint abuse)   | GET /api/health                  | accept      | Rate limiting deferred to Phase 6 (noted in CONTEXT.md deferred ideas); SELECT 1 is minimal overhead; acceptable for dev starter                                |
| T-05-08   | Tampering (SQL Injection via health check)  | client`SELECT 1`                 | accept      | Query is a hardcoded literal string — no user input in the query; tagged template with no interpolation; no injection surface                                   |
| T-05-09   | Information Disclosure (DB client exposure) | src/lib/server/db/index.ts       | mitigate    | client is exported from $lib/server/ — SvelteKit prevents client-side imports at build time; raw postgres client is not reachable from browser bundles          |

</threat_model>
