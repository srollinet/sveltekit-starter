**Prerequisites:** Ensure PostgreSQL is running. If not already up:

```bash
docker compose up -d postgres
```

Wait for the health check to pass:

```bash
docker compose exec postgres pg_isready -U postgres -d app
```

**Step 1 — Push schema to database (per D-09, dev workflow):**

```bash
npx drizzle-kit push
```

This syncs the posts table schema directly to the running PostgreSQL instance. If drizzle-kit prompts for confirmation, the prompt should auto-accept in non-TTY mode. If it blocks, try:

```bash
echo "yes" | npx drizzle-kit push
```

Or if that fails, use the `--force` flag if available. The push must complete successfully.

**Step 2 — Verify the table exists in PostgreSQL:**

```bash
docker compose exec postgres psql -U postgres -d app -c "\dt posts"
docker compose exec postgres psql -U postgres -d app -c "\d posts"
```

Expected: `posts` table with columns: id (uuid), title (text), body (text), status (post_status enum), created_at (timestamptz), updated_at (timestamptz).

**Step 3 — Generate the first migration SQL file (per D-12):**

After the push, the database already has the schema. To generate the migration SQL for the committed `drizzle/` directory, run:

```bash
npx drizzle-kit generate
```

This produces files in `drizzle/`:

- `drizzle/0000_*.sql` — the SQL migration file (name may vary, e.g., `0000_initial.sql` or `0000_some_name.sql`)
- `drizzle/meta/_journal.json` — migration journal
- `drizzle/meta/0000_snapshot.json` — schema snapshot

NOTE: The migration file name is auto-generated by drizzle-kit. The exact name (e.g., `0000_initial.sql` vs `0000_loud_doctor.sql`) is determined by drizzle-kit. Accept whatever name it generates.

**Step 4 — Verify the generated migration SQL contains expected statements:**

The SQL file should contain:

- `CREATE TYPE "public"."post_status" AS ENUM('draft', 'published', 'archived');`
- `CREATE TABLE "posts"` with all six columns
- Column types: `uuid`, `text`, `"post_status"`, `timestamp with time zone`

**Step 5 — Add `drizzle/` to the project by ensuring it is NOT in .gitignore.** The migration files must be committed to git per D-11.

Verify the generated migration can apply cleanly by testing the `db:migrate` workflow against a reset database.

**Step 1 — Drop and recreate the database to test from scratch:**

```bash

docker compose exec postgres psql -U postgres -c "CREATE DATABASE app;"
```

**Step 2 — Apply migration using the production workflow (per D-09):**

```bash
pnpm run db:migrate
```

This should apply the `drizzle/0000_*.sql` migration and create the posts table + post_status enum from zero.

**Step 3 — Verify table exists after migration:**

```bash
docker compose exec postgres psql -U postgres -d app -c "\dt posts"
docker compose exec postgres psql -U postgres -d app -c "SELECT column_name FROM information_schema.columns WHERE table_name='posts' ORDER BY ordinal_position" -t
```

Expected output should list all six columns: id, title, body, status, created_at, updated_at.

**Step 4 — Run the full quality gate:**

```bash
pnpm run check && pnpm run lint && pnpm run knip
```

All three must pass with the new drizzle/ files present.
