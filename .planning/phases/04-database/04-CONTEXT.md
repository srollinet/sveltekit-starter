<decisions>
## Implementation Decisions

### Scaffolding Approach

- **D-01:** Use `npx sv add drizzle="database:postgresql+client:postgres.js"` to scaffold the initial Drizzle setup (without `docker:yes` — Docker is already configured from Phase 2). This is the official Svelte CLI add-on.
- **D-02:** After scaffolding, restructure the generated single `schema.ts` into a `src/lib/server/db/schema/` directory pattern (one file per domain area). This matches DB-03 requirement and demonstrates the scalable pattern.

### Example Table: `posts`

- **D-03:** The example table is named `posts`.
- **D-04:** Primary key: **UUIDv7** — time-ordered UUID, sortable by creation time. Requires the `uuidv7` npm package for generation in `$defaultFn`. Column type: `uuid` in Drizzle.
- **D-05:** Rich columns to demonstrate common patterns:
  - `id` — uuid, primary key, UUIDv7 generated by default
  - `title` — text, not null
  - `body` — text (nullable — drafts may have no body)
  - `status` — Drizzle enum: `'draft' | 'published' | 'archived'`
  - `created_at` — timestamp with timezone, not null, default now()
  - `updated_at` — timestamp with timezone, not null, default now() (updated via trigger or app logic)
- **D-06:** Schema lives at `src/lib/server/db/schema/posts.ts`. `drizzle.config.ts` points to `src/lib/server/db/schema/`.

### DB Client Singleton

- **D-07:** Follow what `sv add drizzle` generates for `src/lib/server/db/index.ts` as the base pattern. If it creates a single-file client with postgres.js + drizzle() export, use that as-is (adjusted for our schema/ directory structure).
- **D-08:** The DB client reads `DATABASE_URL` from the existing `src/lib/server/env/` Zod-validated env — do NOT import directly from `$env/dynamic/private` in db/index.ts. Import from `$lib/server/env` to reuse the validated value.

### Migration Strategy

- **D-09:** Primary workflow:
  - **Dev:** `db:push` — syncs schema directly, no SQL files generated, fast iteration
  - **Prod:** `db:generate` + `db:migrate` — generates SQL files committed to `drizzle/`, applied explicitly
- **D-10:** All four drizzle-kit scripts in `package.json`: `db:generate`, `db:migrate`, `db:push`, `db:studio`
- **D-11:** Migration files live in `drizzle/` at project root (`drizzle/meta/` for snapshots, `drizzle/*.sql` for SQL files). This is the Drizzle default and matches project conventions.
- **D-12:** Phase 4 generates and commits the first migration SQL file (DB-05). This demonstrates the workflow and ensures `drizzle/` directory exists in the starter with a real migration.
- **D-13:** No auto-migration on startup — migrations are always run manually with `pnpm run db:migrate`. This avoids production race conditions and keeps the workflow explicit.

### DB-01 Status

- **D-14:** DB-01 (PostgreSQL 17 in docker-compose.yml with named volume + health check) was completed in Phase 2. Phase 4 does NOT modify docker-compose.yml. Drizzle connects to the already-configured PostgreSQL instance via `DATABASE_URL`.

### Claude's Discretion

- Exact `drizzle.config.ts` contents — follow what `sv add drizzle` generates, adjusted for `schema/` directory
- Whether `updated_at` uses a Drizzle `$onUpdate` or relies on the application to set it (Drizzle ORM v0.28+ has `.$onUpdateFn()` support)
- Specific Drizzle table definition style (using `pgTable` with typed columns)

## Existing Code Insights

### Reusable Assets

- `src/lib/server/env/index.ts` — exports validated `env` object; `env.DATABASE_URL` is available for the DB connection
- `docker-compose.yml` — PostgreSQL 17 running on port 5432 with credentials from `.env.development`

### Established Patterns

- `$env/dynamic/private` is used via the validated env module — DB client follows same pattern
- pnpm is the package manager (`pnpm add`, not `npm install`)
- TypeScript config files preferred (e.g., `knip.config.ts`) — use `drizzle.config.ts` not `drizzle.config.js`
- Server-only code in `src/lib/server/` — Drizzle client singleton belongs there

### Integration Points

- Phase 5 (Observability): OTEL auto-instrumentation for postgres.js will pick up the DB client automatically — no changes to DB layer needed
- Phase 6 (Security): No impact on database layer
- Phase 7 (Testing): `src/lib/server/db/index.ts` will be imported in DB-related unit tests

<specifics>
## Specific Ideas

- Use UUIDv7 for primary keys via the `uuidv7` npm package — sortable by time, better than UUIDv4 for database indexes
- `sv add drizzle` is the official Svelte CLI integration path — use it for scaffolding rather than manual setup
- The `posts` table should feel like a real starting point, not an obviously fake placeholder — rich enough to demonstrate enum types and optional fields
